# Order and Payment Management API

A Laravel 12 REST API for user authentication, order management, and payment processing. The application is organized into modules for Users, Orders, and Payments, with JWT authentication and a gateway-based payment design.

## Project Overview

This project implements:

- User registration, login, logout, and authenticated profile retrieval.
- JWT-protected API endpoints under `/api/v1`.
- Order creation, listing, status filtering, updating, and deletion.
- Payment processing for confirmed orders.
- Multiple payment gateway implementations selected through a factory.
- Consistent JSON API responses.
- Route-based API documentation generated by Scribe, including a Postman collection.

## Architecture & Design

The application uses a modular structure powered by `nwidart/laravel-modules`.

Implemented modules:

- `Users`
- `Orders`
- `Payments`

The project uses a service layer. Controllers handle HTTP requests and responses, while services contain the main business logic. The code does not implement a repository pattern; services work directly with Eloquent models and Laravel database transactions.

This structure keeps each domain area isolated and makes the payment logic easier to extend without changing the controllers.

### Layers

- **Controllers**: Receive API requests, call services, and return standardized JSON responses.
- **Services**: Contain business logic for authentication, orders, and payments.
- **Models**: Eloquent models for `User`, `Order`, `OrderItem`, and `Payment`.
- **Resources**: Transform order and payment models into API response arrays.
- **Requests**: Validate input and return the project JSON error format on validation failure.
- **Gateways**: Payment gateway strategies implementing a shared interface.
- **Tests**: Root Laravel example tests plus module feature/unit tests for authentication, orders, payments, and gateways.
- **Repositories**: No repository layer is implemented in the current codebase.

## Authentication

Authentication is implemented with `tymon/jwt-auth`.

The `UsersServiceProvider` registers an `api` guard dynamically:

```php
'driver' => 'jwt',
'provider' => 'users',
```

The module user model implements `JWTSubject`.

### Auth Flow

1. Register or login.
2. Read the returned token from `data.authorization.token`.
3. Send the token with protected requests:

```http
Authorization: Bearer <token>
```

### Auth Endpoints

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
POST /api/v1/auth/logout
GET  /api/v1/auth/me
```

`logout` and `me` require JWT authentication.

## Order Management

Order endpoints are JWT-protected.

```text
GET    /api/v1/orders
POST   /api/v1/orders
PUT    /api/v1/orders/{id}
DELETE /api/v1/orders/{id}
```

### Implemented Features

- Create an order with one or more items.
- Calculate `total_amount` from item quantity and price.
- Store order items in the `order_items` table.
- List authenticated user's orders with pagination.
- Filter orders by `status`.
- Update order `status` and `total_amount`.
- Delete an order if it has no payments.

### Order Statuses

```text
pending
confirmed
completed
cancelled
```

### Business Rules

- New orders are created with `pending` status.
- Orders are created inside a database transaction.
- An order cannot be deleted if it has any payments.
- The order list only returns orders for the authenticated user.

Example create order payload:

```json
{
  "items": [
    {
      "product_name": "Keyboard",
      "quantity": 2,
      "price": 49.99
    }
  ]
}
```

Example filter:

```text
GET /api/v1/orders?status=confirmed
```

## Payment System

Payment endpoints are JWT-protected.

```text
GET  /api/v1/payments
POST /api/v1/payments/process
```

### How Payment Processing Works

`PaymentService::processPayment()` performs the payment flow:

1. Starts a database transaction.
2. Loads the order using `lockForUpdate()`.
3. Rejects payment unless the order status is `confirmed`.
4. Resolves the gateway with `PaymentGatewayFactory`.
5. Calls the selected gateway's `charge()` method.
6. Creates a payment record.
7. If the gateway returns success, updates the order status to `completed`.

Payments can be listed for the authenticated user and optionally filtered by `order_id`.

```text
GET /api/v1/payments?order_id=1
```

### Payment Methods

The current code supports these payment method constants:

```text
credit_card
paypal
paymob
```

### Payment Statuses

```text
pending
successful
failed
```

### Gateway Design

Payment gateways implement:

```php
Modules\Payments\Gateways\Contracts\PaymentGatewayInterface
```

The interface requires:

```php
public function charge(float $amount, array $details): array;
```

Existing gateways:

- `CreditCardGateway`
- `PaypalGateway`
- `PaymobGateway`

`CreditCardGateway` and `PaypalGateway` return successful mocked transaction responses with generated transaction IDs.

`PaymobGateway` returns a successful mocked response with a generated transaction ID and a sandbox checkout URL. It reads Paymob config from `config('payments.gateways.paymob.config')`.

## How to Add a New Payment Gateway

To add a gateway using the current implementation:

1. Create a new gateway class under:

```text
Modules/Payments/app/Gateways
```

2. Implement the existing interface:

```php
use Modules\Payments\Gateways\Contracts\PaymentGatewayInterface;

class NewGateway implements PaymentGatewayInterface
{
    public function charge(float $amount, array $details): array
    {
        return [
            'success' => true,
            'transaction_id' => 'NEW_123',
        ];
    }
}
```

3. Register the gateway in:

```text
Modules/Payments/config/config.php
```

Example:

```php
'gateways' => [
    'new_gateway' => NewGateway::class,
],
```

or, if it needs configuration:

```php
'gateways' => [
    'new_gateway' => [
        'driver' => NewGateway::class,
        'config' => [
            'key' => env('NEW_GATEWAY_KEY'),
        ],
    ],
],
```

4. Add the new method to the validation rule in:

```text
Modules/Payments/app/Http/Requests/ProcessPaymentRequest.php
```

The factory already reads from `config('payments.gateways')`, resolves the configured class from the container, and verifies that it implements `PaymentGatewayInterface`.

## API Design

The API follows REST-style routes and JSON responses.

### Route Prefix

```text
/api/v1
```

### Implemented HTTP Status Codes

- `200`: Successful read, update, delete, login, logout, and payment processing.
- `201`: Successful registration and order creation.
- `401`: Invalid credentials or unauthenticated protected request.
- `404`: Order not found in order update/delete or payment processing.
- `422`: Validation failure or business rule violation.

### Pagination

Order and payment list endpoints use Laravel pagination through service methods:

- `OrderService::paginateForUser()`
- `PaymentService::paginateForUser()`

The API response includes pagination metadata when a resource collection wraps a `LengthAwarePaginator`.

Example:

```json
{
  "success": true,
  "message": "Orders retrieved successfully.",
  "data": [],
  "meta": {
    "current_page": 1,
    "last_page": 1,
    "per_page": 15,
    "total": 0
  }
}
```


```xml

<env name="DB_DATABASE" value=":memory:"/>
```

## Additional Notes

- Payment gateway implementations are currently mocked/simulated. They generate transaction IDs and do not call real external payment APIs.
- `PaymobGateway` builds a sandbox checkout URL from local config and request details.
- Duplicate successful payment attempts are prevented indirectly because a successful payment changes the order status to `completed`, and the payment service only accepts orders with `confirmed` status.
- There is no authorization policy limiting order update/delete by owner in the current controller/service implementation; the list/create flows use the authenticated user's ID.
- Scribe is configured to generate static documentation and a Postman collection under `public/docs`.
- OpenAPI generation is disabled in the Scribe config.
